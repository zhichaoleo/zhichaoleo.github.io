设计模式读书笔记(1)

发表于 2016-11-28   |   分类于 技术，代码优化   |     |   阅读次数   |  

一切为了：可维护，可扩展，可服用，灵活性好。强内聚，松耦合。

基本设计原则总结

1.单一职责原则

单一职责原则(SRP):就一个类而言，应该仅有一个引起它变化的原因
-ASD

软件设计真正要做的许多内容，就是发现职责并把那些职责互相分离[ASD]。其实，要分辨是否应该分离出类来也简单，那就是：
如果你能够想到多于一个动机去改变一个类，那么这个类就具有多于一个的职责[ASD]。

自我总结：

我们在OOP编程最初特别容易犯一种错误，就是建一个类，然后把各种属性，各种行为函数往里面塞，这样在初期代码规模较小时没什么大问题，可以用，但是到后期，代码的维护，修改变得几乎不可行。可谓牵一发而动全身，标准的—>弱内聚，强耦合。
我们在一开始设计整个程序的时候就应该考虑到这一点，划分好职责，通过UML类图去看，推敲，心里要明白各个类是负责什么的，相互关系又是什么，最好的办法是画好UML类图，直观的去看。

2.开放-封闭原则

开放-封闭原则，是说软件实体(类，函数，模块等等)应该可以扩展，但是不可以修改。[ASD]

一句话包含了两层意思：

1，对于扩展是开放的（Open for extension）

2, 对于更改是封闭的（Closed for modification）

如果存在能让一个程序员给你拼命的事，一是你融了他的装备，二是改需求。面对不断更改的需求，牵一发动全身式的代码显然不行。问题就是：怎样的设计才能面对需求的改变却可以保持稳定，从而使得系统在第一个版本以后不断推出新的版本呢？
答案就是要严格遵守： 开放-封闭原则

面对需求，对程序的修改是通过增加新代码完成的，而不是改现有代码。开放-封闭的意思就是，在设计类的时候，时刻要考虑，尽量让这个类足够好，写好了，就不去改了，如果新需求来了，我们增加一些新的类就完事儿了，原来的代码能不动就不动。

实际上，这个很难做到，无论模块多么封闭，都会有一些无法对之封闭的方法，我们在遇到变化的时候要 ASAP 立即采取行动，利用抽象类来隔离那些变化。

自我总结：

最好是在开发开始后不久就可以知道可能会发生的变化，增加的需求。当然最好是不改需求。。。因为，查明可能发生的变化所等待的时间越长，要创建正确的抽象就越难。

开放-封闭原则是OOP体现其价值的核心所在，遵循它就可以得到OOP的好处，

可维护，可扩展，可服用，灵活性好。

但是，我们不能瞎抽象，对于程序中的每个部分都刻意的进行抽象是不理智的。 拒绝不成熟的抽象，和抽象本身一样重要——ASD

3.依赖倒转原则

里氏代换原则(LSP):子类型必须能够替换掉他们的父类型。[ASD]

一个软件实体如果使用的是一个父类的话，那么一定适用于其子类，而且它觉察不出父类对象与子类对象的区别。也就是说，把代码里的父类对象全部换成子类对象，程序也一样运行，程序的行为没有任何改变。

只有当子类可以替换掉父类，软件单位的功能不收到影响时，父类才能真正被复用，而子类也能够在父类的基础上增加新的行为。

>依赖倒转原则：抽象不应该依赖细节，细节应该依赖于抽象（接口或者抽象类）。其实就是，要针对借口编程，而不是针对实现编程。

依赖倒转原则：

A. 高层模块不应该依赖低层模块，二者都应该依赖于抽象。

B. 抽象不应该依赖于细节，细节应该依赖于抽象。

自我总结：

正是有了Barbara Liskov女士提出的里氏代换原则，才使得开放-封闭得以实现，由于子类型的可替换性才使得我们可以在不修改父类的情形下，通过子类型进行扩展。

如果不管高层模块还是低层模块都是依赖于抽象（抽象类或接口）那么，只要抽象是稳定的，那么对各个模块的任何更改都能确保不会影响到其他模块，这样就使得无论高层模块还是低层模块都可以被轻易地复用。

依赖倒转可以被看作是OOP的标志，不局限于语言的具体实现，如果编写程序时，考虑的都是如何针对抽象编程而不是针对细节编程，即程序中所有的依赖关系都终止于抽象类或者接口，那么就可以说是面向对象程序设计，反之就是面向过程的。
